# 《五行行者》开发规则

这份文档用于整理当前项目已经明确的开发约定，减少后续线程反复踩坑。后续默认以本文件为准；如果用户有新的明确要求，新要求优先，并应及时回写本文件。

## 1. 总体原则

- 项目当前阶段重点是“收口”和“表现层精修”，不是继续无节制铺新系统。
- 开发顺序优先保证：主流程稳定、UI 可视化可调、配置驱动可扩展。
- 能直接做就直接做，少来回确认，除非存在明显分叉风险。
- 用户对 UI 排版、中文显示、乱码、重复劳动非常敏感，改动时要优先规避这些问题。

## 2. UI 开发规则

- 全项目 UI 使用 `UGUI`，不使用 `UIToolkit`。
- 页面和弹窗统一走 `prefab` 驱动，不要长期依赖运行时代码临时拼装界面。
- 开发思路优先 `prefab-first`：
  - 结构、布局、视觉优先改 prefab
  - 脚本主要负责逻辑、绑定、播放和状态同步
- 重建 UI 统一通过 Unity 菜单：`工具/重建UI预设`。
- 如果涉及通用布局结构改动，优先看并修改：
  - `Assets/Editor/UIPrefabBuilder.cs`
- 已明确尽量避免生成器反复覆盖：
  - `BattlePage.prefab` 不要轻易再塞回自动重建链
- 不同 UI 领域尽量分开处理：
  - 不要把剧情/对话 UI 和战斗 UI 混着改
  - 战斗表现优先收在战斗页内部，不随便扩散到别的模块
-备份图，多余图，设计稿放在根目录Arts文件夹
## 3. 配置与数据规则

- 源配置统一放在 `Docs` 目录下，运行时读取 `JSON`，不再走 `.asset` 配置。
- 当前配置链路采用 `CSV -> JSON -> Resources`。
- 配置导入、自动同步、校验应保持可用，尽量不要退回手工导表。
- 如果需要新增固定文案、标点、前后缀、提示文本，优先进入 `Localization.csv`。
- 配置表变更后要注意：
  - 自动同步是否正常
  - 校验是否仍通过
  - 导出结果是否产生无意义 diff

## 4. 文案与编码规则

- 编辑器默认中文，运行时默认中文，同时保留中英文切换。
- 代码里尽量少写中文常量，尤其是长句、特殊标点、高频复用文本。
- 容易乱码、容易复用、带标点的文本优先放进多语言表。
- 项目历史上多次出现中文编码导致脚本编译错误，后续继续遵循：
  - 逻辑代码尽量保留 ASCII key
  - 中文展示尽量交给本地化系统

## 5. 流程与玩法规则

- 当前主流程为：
  - `开始页 -> 地图 -> 节点 -> 战斗/事件 -> 结算 -> 回地图`
- `MainMenuPage` 已退出主流程，开始页直接进地图。
- 地图当前先做一维地图，后续副本再考虑升级为二维。
- 地图移动会推进时间，寿元耗尽则本轮结束。
- 阳寿显示口径采用古风纪年，例如：
  - `阳寿：余二十九载四月`
- 当前关卡允许反复挑战，用于刷成长和装备。
- 第 1 关当前是“新手村”占位，暂不做正式战斗内容。

## 6. 战斗规则

- 战斗系统当前已走配置驱动，技能效果、公式、元素关系、敌人编组均已接通。
- 战斗页支持每方最多 6 个角色，采用 `3 x 2` 排列。
- `BeforeBattle` 在战斗页内、正式开打前触发。
- `AfterBattle` 在战斗结束后、奖励或失败跳转前触发。
- 战斗表现层当前优先级较高，但要遵守：
  - 第一优先尽量改 `UIBattlePage.cs`
  - 先做表现，不轻易改战斗核心结构
  - 如果后续要更准的表现，再补结构化战斗事件字段
- 明确禁止：
  - 不要再引入测试性隐藏数值补正

## 7. 地图与交互规则

- 地图页必须走 prefab，不再回退到运行时工厂版地图页。
- 地图页当前要求重点是“少挤、少重复、少滚动”。
- 用户已经明确的方向包括：
  - 节点信息框和个人信息框并排
  - 简报框去掉
  - 重复信息只保留一份
  - 滚动条尽量去掉
  - 避免“左边太挤、右边太空”
- 地图移动结束后需要明显的到达提示。

## 8. 结算与弹窗规则

- 战斗胜利：
  - 主按钮应为“下一关”或等价主推进语义
  - 次按钮为“返回地图”或等价语义
- 战斗失败：
  - 不要保留“再战”“返回地图”这类普通选项
  - 应偏“本轮失败/游戏结束/返回标题”语义
  - 失败弹窗不要显示胜利专属信息块
- 奖励弹窗、功法弹窗、装备弹窗要尽量各司其职，不混用旧弹窗语义。

## 9. 装备与功法规则

- 装备系统已经从模板 ID 逐步升级到实例化思路，后续继续按实例推进。
- 同名装备不能简单叠加处理，因为后续要支持随机属性。
- 装备 UI 应尽量朝这类结构收口：
  - 固定槽位
  - 候选区
  - 详情区
- 功法已支持等级、品质和多效果描述。
- 功法品质当前更适合用颜色表达，不再强调“品质文字”。

## 10. 日志与工程输出规则

- 不希望在项目根目录散落各种日志文件。
- 额外日志尽量放到 `Logs` 目录。
- `.gitignore` 已处理常见日志和 Burst 调试目录，保持这个约定。

## 11. 当前推荐工作方式

- 开始新任务前，先看相关 prefab、脚本、配置是否已经有现成结构，不要凭印象乱改。
- 改 UI 时，优先确认 prefab 结构和脚本绑定是否一致。
- 改配置时，顺手检查自动同步和本地化文本是否被破坏。
- 改表现层时，优先做一版可用方案，再逐步加精度，不要一开始做成大重构。
- 若用户新要求已经稳定，应及时把规则沉淀回本文档。

## 13. 会话与协作约定

- 默认全程用中文沟通。
- 每次聊天会话结束，或者遇到需要决策时，播放一次 Windows 系统提示音。
- 能安装到 `D` 盘的软件尽量不要装到 `C` 盘。
- 决策提醒也用中文表达。
- 能修改prefab的就不要写入代码，用户需要直观在prefab里调整

## 14. 更新规则

- 当用户明确改变方向时，应优先执行新要求。
- 新要求如果会长期影响开发方式，应同步更新本文件。
- 若本文件与更晚的交接文档冲突，以时间更新、更明确、用户直接确认过的规则为准。
