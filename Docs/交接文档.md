# 《五行行者》项目交接文档

## 1. 项目概况
- 项目名称：`五行行者`
- 项目类型：Unity 单机肉鸽原型
- 当前定位：先验证单局闭环、地图推进、寿元系统、装备成长、自动战斗，不优先打磨美术
- 当前技术路线：全项目 UI 使用 UGUI，页面走 prefab 驱动，配置源数据放在 `Docs` 下 CSV，再转成 `Assets/Resources` 下 JSON 供运行时读取

## 2. 我们已经明确约定的规则

### 2.1 UI 与工程规则
- 全项目 UI 使用 UGUI，不做 UIToolkit
- 页面和弹窗统一走 prefab 驱动，不走纯运行时代码临时拼装
- 地图页也必须走 prefab，不再使用运行时工厂版地图页
- 重建 UI 统一通过 Unity 菜单：`工具/重建UI预设`
- 以后动到布局结构时，应优先改 `Assets/Editor/UIPrefabBuilder.cs`，然后重建 prefab

### 2.2 配置规则
- 源配置都放在 `Docs` 下，当前已有：
  - `Character.csv`
  - `Enemy.csv`
  - `Skill.csv`
  - `Equipment.csv`
  - `Localization.csv`
- 运行时读 JSON，不再使用 `.asset` 配置
- JSON 自动同步，不希望每次手动点导入按钮
- 语言表和其他配置同样走 CSV -> JSON
- 如果需要新增标点、前后缀、固定提示文案，优先放进 `Localization.csv`

### 2.3 文案与编码规则
- 编辑器默认中文，运行时默认中文
- 多语言必须保留，支持中文/英文切换
- 代码里尽量少写中文，尤其少写易出乱码的中文常量
- 容易重复、容易乱码、带标点的文案优先进多语言表
- 项目曾多次遇到中文编码导致脚本报错，因此后续线程应继续把高风险中文常量往语言表迁移

### 2.4 玩法规则
- 地图当前先做一维地图，后续做副本后再升级成二维地图
- 地图每走一步会推进时间
- 时间显示采用古风寿元纪年：`阳寿：余二十九载四月`
- 到达寿元尽头则本轮结束
- 当前关卡允许反复挑战，用于刷成长、刷装备，再推进后续关卡
- 第 1 关不是战斗，作为“新手村”占位，后续再补事件/功能

### 2.5 日志与输出规则
- 不想在项目根目录看到各种日志文件
- 额外日志应尽量放到 `Logs` 文件夹
- `.gitignore` 已补充常见日志和 Burst 调试目录忽略规则
- `codexDlablo_BurstDebugInformation_DoNotShip` 不是手工加的，是 Unity/Burst 构建自动生成，已加入 gitignore

## 4. 当前流程逻辑（非常重要）

### 4.1 主流程
当前期望流程：
1. 主菜单
2. 地图页
3. 选择/进入节点
4. 若为战斗节点，则移动结束后先显示到达 Toast，再延迟自动开战
5. 战斗结束后根据胜负回流
6. 继续地图推进或结束本轮

### 4.2 地图页当前期望行为
- 至少显示当前关、前一关、后一关（如果存在）
- 节点用按钮呈现，用线连接
- 可点击查看节点信息
- 有前进、后退、进入节点按钮
- 移动应有大约半秒动画
- 动画结束自动触发目标节点事件
- 第 1 关是新手村占位，先空着不打战斗
- 地图页要提供查看装备入口
- 战斗前的简报内容已经要求尽量前置到地图页里

### 4.3 战斗页当前期望行为
- 进入战斗时不需要额外“准备界面”停太久
- 但自动开战前需要留出足够时间看清 Toast，并给逃跑反应时间
- 当前要求：自动开战延迟应跟 Toast 的显示时长同步，而不是固定 1 秒
- 战斗页应保留逃跑按钮
- 点击逃跑：终止当前战斗，给明显提示，返回前一关

### 4.4 战斗结算当前约定
胜利：
- 主按钮：`下一关`
- 次按钮：`返回地图`
- 点击下一关也要出现和移动到达类似的提示
- 胜利弹窗里的“战利品”和“下一步”两行，用户后续可能继续精简

失败：
- 取消按钮已明确要求去掉
- 不要出现“再战”/“返回地图”这类选项
- 只保留一个回主界面的结束按钮
- 文案要偏“游戏结束，请重新开始或重置本轮”这种语义
- 用户明确要求：战利品和下一步这两行不要出现在失败弹窗里

## 5. 最近几轮用户明确提出的界面要求

### 5.1 地图页
用户最近的明确要求如下：
- `耗时 1步` 改成 `耗时 xx月`
- 地图页面里多余内容删掉，减少占地
- 节点信息框和个人信息框并排
- 简报框去掉
- 节点详情和节点说明意思重复时，只保留一套
- 耗时重复时，只保留一处
- 滚动条尽量去掉
- 当前排版存在“左边太挤、右边太空”的问题，考虑改成双栏/双排重新布局
- 地图移动结束后，增加显眼的关卡飘字提示
- 飘字要放屏幕中间，更显眼，可调整颜色

### 5.2 战斗页
用户最近的明确要求如下：
- 进入战斗前，停留时间要足够看完 Toast
- 自动开战延迟应跟随 Toast 时长
- 需要保留逃跑按钮
- 逃跑后应回前一关
- 战斗失败的取消按钮去掉
- 失败时要更像游戏结束而不是普通确认框
- 胜利时关闭按钮改成“返回地图”
- 胜利主按钮是“下一关”
- 失败不要“再战”和“返回地图”
- 失败只保留“返回主界面”或同义语义的按钮
- `战利品` 和 `下一步` 两行不要出现在失败弹窗里

### 5.3 装备页
用户最近的明确要求如下：
- 背包和当前穿戴不能混在一起显示
- 必须能明显地区分“当前身上装备”和“背包可选装备”
- 装备列表必须是动态的，不要写死显示 4 条
- 同名装备不要叠加
- 后续要支持随机属性，因此背包必须按实例思路继续演进

## 6. 当前已知问题与高风险点

### 6.1 中文编码问题
这是项目历史上最反复的问题之一。
- 多次出现 `BattleManager.cs` 等文件因为中文乱码直接编译报错
- 已做一些缓解措施，但还没有彻底根治

后续线程建议：
- 继续把高风险中文字符串迁入 `Localization.csv`
- 代码里尽量保留 ASCII key，不直接硬编码复杂中文句子
- 尤其避免在逻辑代码里写长中文段落和特殊标点

### 6.2 地图 prefab 与代码耦合较高
- `UIMapPage.cs` 与 `UIPrefabBuilder.cs` 最近改动频繁
- 很多布局需求是用户基于截图反馈逐步收的
- 后续每次改地图布局，都应先核对 prefab 结构，再改脚本绑定

### 6.3 装备实例化尚未彻底完成
- 现在背包已支持重复条目，不再简单去重
- 但真正意义上的“装备实例 ID + 随机词条”还没正式落地
- 这是后续装备系统最应该补的底层能力之一

### 6.4 战斗与地图的切换时序仍可能需要细调
- Toast 时长、自动开战延迟、逃跑反应时间，目前是按用户反馈临时调优
- 后续可能还要继续根据实际手感细调

## 7. 建议新线程接手后的优先顺序

### 第一优先：稳定地图页排版

- 让按钮区、节点图区、信息区更清晰分层

### 第二优先：稳定战斗结算交互
- 再核对胜利/失败弹窗按钮是否完全符合最新约定
- 尤其确认失败时是否真的只有一个按钮
- 再确认失败文案是否已经改成“游戏结束，请重新开始/重置本轮”方向

### 第三优先：装备实例化
- 给背包装备引入实例 ID
- 为后续随机词条做准备
- 让装备列表真正区分“同模板不同实例”

### 第四优先：新手村内容补位
- 第 1 关当前只是占位节点
- 后续可以在这里接教学、商店、初始选择、简易事件等

## 9. 给下一个线程的简短接手建议
如果新线程要直接继续开发，建议先做这 3 件事：
1. 先通读地图页、战斗页、进度管理这 3 个脚本，确认最新交互是否已全部兑现。
