# 第二阶段拆解

## 阶段目标

第二阶段的目标，是在第一阶段自动战斗原型已经跑通的基础上，打通真正可反复验证的单局闭环。

完成标准：

1. 可以从主菜单进入地图页
2. 可以在地图上推进节点并触发对应事件
3. 战斗结束后可以正确回流到地图
4. 战斗胜负、撤退、寿尽都能给出明确结算
5. 可以获得局内奖励，并将结果写回当前局数据

---

## 工作范围

第二阶段重点覆盖四块内容：

1. 地图推进流程
2. 战斗结算与奖励回流
3. 功法/技能奖励选择
4. 单局数据与流程闭环

不在第二阶段内的内容：

- 二维地图或复杂分叉地图
- 大量章节内容扩展
- 完整随机词缀装备体系
- 局外养成与长线存档
- 大规模美术替换与特效表现

---

## 任务拆解

### E. 建立地图推进流程

目标：

- 让玩家从“单场战斗验证”进入“地图驱动推进”

子任务：

1. 建立地图节点数据结构
2. 支持至少一维线性地图推进
3. 支持基础节点类型：`Battle / Elite / Rest / Boss`
4. 展示当前关、前一关、后一关等核心信息
5. 支持进入节点、前进、后退、重置本轮
6. 接入节点触发后的页面跳转或事件处理
7. 接入章节或区域命名，用于流程展示

建议第一版规则：

- 第一版地图采用线性推进
- 每个节点都应有名称、类型、说明、耗时
- 地图页优先保证信息可读和按钮路径清晰
- 第 1 关可作为新手村或占位节点，先不强行接战斗

验收标准：

- 玩家可以在地图上稳定推进
- 不同节点进入后能触发正确逻辑
- 地图与战斗之间的回流关系明确

---

### F. 接入战斗结算与奖励

目标：

- 让战斗不再只是“打完即结束”，而是能影响当前局进度

子任务：

1. 明确战斗胜利、失败、撤退三种结算分支
2. 胜利后结算灵石、经验、装备或其他局内奖励
3. 失败后给出本轮结束或回主界面的明确结果
4. 撤退时支持回退到上一关或执行预定惩罚
5. 战斗结果弹窗与按钮语义保持一致
6. 将奖励结果写入当前局数据
7. 地图回流后刷新主界面与地图状态

建议第一版规则：

- 胜利后允许继续推进
- 失败直接结束当前局，避免过多岔路
- 撤退优先作为保底交互，不做复杂概率判定
- 奖励先保证“拿到并生效”，再考虑表现包装

验收标准：

- 一场战斗结束后，玩家能明确知道发生了什么
- 奖励会真实影响后续流程
- 胜利、失败、撤退三条路径都不会卡死流程

---

### G. 实现功法奖励选择第一版

目标：

- 让局内成长不只体现在数值，还能体现构筑方向

子任务：

1. 建立战后奖励池生成逻辑
2. 支持三选一功法或技能奖励界面
3. 将选中的功法写入当前局角色数据
4. 支持已有技能升级或新增技能两种结果
5. 将五行分类接入奖励展示
6. 处理奖励放弃、奖励已满、重复奖励等边界情况

建议第一版规则：

- 每次胜利后固定弹出三选一
- 奖励池可先按简单权重随机
- 第一版先覆盖主动技能、治疗、防御、DOT、控制等常见效果
- 先不做过深的灵根限制和复杂派生规则

验收标准：

- 玩家可以稳定完成一次奖励选择
- 选择结果会进入当前局战斗数据
- 下一场战斗能体现奖励选择的影响

---

### H. 打通单局数据闭环

目标：

- 让地图、战斗、奖励、寿元消耗都写回同一份当前局数据

子任务：

1. 建立或补齐 `RunData`、章节进度、奖励记录等结构
2. 记录当前节点、已推进月数、剩余寿元、胜负状态
3. 记录当前拥有角色、技能、装备、灵石等局内资源
4. 将地图推进、战斗结果、奖励选择统一写回
5. 处理本轮结束、重置本轮、返回主菜单等流程
6. 为后续存档和局外系统预留数据入口

建议第一版规则：

- 单局数据优先集中管理，避免分散在多个页面各自保存
- 重置本轮时应能回到干净起点
- 主菜单应能读到本轮核心摘要信息
- 寿元消耗应作为地图推进的统一资源进行结算

验收标准：

- 玩家完整跑完一局核心流程时，数据不会丢失或错乱
- 地图、战斗、奖励三者使用的是同一份当前局状态
- 本轮结束与重置逻辑清晰可验证

---

## 推荐执行顺序

按下面顺序做，返工会最少：

1. 先打通地图推进
2. 再接战斗结算回流
3. 再接三选一功法奖励
4. 最后统一收口当前局数据

原因：

- 没有地图推进，第二阶段就没有主干
- 结算不回流，战斗仍然只是孤立页面
- 奖励选择必须建立在“战斗打完还能继续推进”之上
- 单局数据最好在主要流程确定后统一收口

## 风险点

### 风险 1：地图页与 prefab 结构耦合过高

影响：

- 地图布局频繁调整时，容易出现脚本绑定失效或显示错位

应对：

- 优先通过统一的 prefab 构建入口调整结构
- 每次改布局时同步核对脚本字段与 prefab 层级

### 风险 2：战斗回流时序不稳定

影响：

- 容易出现 Toast、结算弹窗、自动开战、返回地图之间互相打架

应对：

- 统一战斗结束后的状态流转顺序
- 把延迟、动画、跳页时机放进明确流程节点里管理

### 风险 3：当前局数据分散

影响：

- 地图、战斗、奖励各管一份状态，后续很容易出现同步问题

应对：

- 尽快集中管理单局状态
- 页面只读写统一数据源，不各自保存关键结果

### 风险 4：奖励系统先做重了

影响：

- 会拖慢单局闭环验证，导致第二阶段目标失焦

应对：

- 第一版先保证奖励选择真实生效
- 稀有度、复杂词条、深层规则放到后续阶段

## 第二阶段数值规则补充

为了先把单局节奏和成长反馈做稳，当前版本补充采用以下规则：

### 1. 友方默认单角色

- 当前局默认只上阵 1 名主角，不再默认上阵 2 人
- 后续如果要做召唤物、分身、援护或第二角色，再基于现有结构扩展
- 当前默认主角为 `C001 金锋卫`

### 2. 功法增加等级与品质

- 每个功法拥有 `品质` 字段，当前分为：`普通 / 稀有 / 绝品`
- 每个功法拥有 `等级`，首次获得时为 `Lv.1`
- 再次获得同名功法时，不再重复获得，而是使该功法 `+1 级`
- 功法等级会直接影响战斗中的技能威力，不只是展示文本变化

### 3. 奖励品质权重规则

- 普通战斗更容易掉落 `普通` 功法
- 精英战斗更容易掉落 `稀有` 功法
- Boss 战斗更容易掉落 `绝品` 功法
- 已拥有功法允许继续进入奖励池，用于触发升级

### 4. 当前仍需继续验证的数值点

- 单主角对双敌方时的难度是否过高
- 精英/Boss 品质权重是否过强，导致成长过快
- 同功法升级后的伤害成长曲线是否平滑
- 失败、撤退、寿元耗尽时的数值惩罚是否需要继续补齐
