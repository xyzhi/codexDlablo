# 策划给开发的文档

这份文档以后固定用于记录：
- 策划确认后的开发补充需求
- 当前版本尚未接完、但后续必须补齐的系统能力
- 希望主程优先做成配置化的运行时与工具支持

这份文档是策划写给开发看的固定文档。
如果是开发反向告诉策划“当前哪些字段已接入、怎么配、有什么约束”，统一看：
`Docs/策划配置总说明.md`

## 当前版本背景

- 当前主流程关卡数按 `StageNode.csv` 配置读取
- Boss 关卡位置按 `StageNode.csv` 配置
- 最终首领关由策划在 `StageNode.csv` 中配置
- 战斗公式由 `BattleFormula.csv` 配置读取
- 元素关系与特殊元素标签应由 `ElementRelation.csv` 配置读取
- 敌方编组、元素覆盖、Boss 模板不应写死在代码里，应由策划配置决定
- 当前数值和文案尽量希望继续走配置给策划维护

## 一、技能系统需要改成“技能库全收 + 上场限3 + 条件触发施法”

### 当前策划结论

技能掉落方案采用：
- 掉落技能可以全部收入技能库
- 不做“背包满3个就必须丢旧技能”的方案
- 当前版本暂时不做“遗忘技能换资源”功能
- 角色上场时最多装备 `3` 个主动技能
- 战斗中不采用“按回合固定轮放技能”的机制

### 不采用固定轮放的原因

如果按回合轮放：
- 垃圾技能会强行占行动轴
- 玩家拿到新技能不一定更爽，可能反而把循环打坏
- 自动战斗会显得很蠢
- 构筑策略会退化成“尽量别拿废技能”

因此当前策划希望改成：
- 技能不是轮流放
- 而是每回合从“当前可释放的已装备技能”里，自动选一个最该放的技能

### 目标机制

建议战斗施法流程如下：
1. 每回合角色行动时，先检查已装备的主动技能
2. 筛出当前满足释放条件的技能
3. 按优先级选出一个技能释放
4. 如果没有任何技能满足条件，则普攻

也就是：
- 普攻永远作为兜底动作
- 主动技能通过条件触发和优先级竞争行动机会
- 被动技能/触发技则走独立触发逻辑

### 技能结构建议

建议把技能分成三层：
- 普攻：默认兜底，永远存在
- 主动技能：满足条件时自动释放，角色最多装备 `3` 个
- 被动/触发技：满足指定事件时自动触发，不占主动技能栏

同时建议：
- 不要只靠一个 `EffectType` 字符串分支继续硬扩
- 技能本体与技能效果尽量分表维护，建议固定为 `Skill.csv + SkillEffect.csv`
- 一个技能应允许挂多个效果，不要强制只能单效果

### 技能库与技能栏规则

当前策划希望：
- 玩家可以拥有完整技能库
- 但战斗前只能从技能库里选择最多 `3` 个主动技能上场
- 技能栏调整应放在非战斗节点进行，例如：休整点、洞府、特定事件、准备界面
- 不建议允许战斗中随时换技能

这样玩家的策略重点就会变成：
- 这关该带哪 `3` 个技能
- 打 Boss 前要不要调整技能栏
- 当前构筑更偏输出、续航、控制还是保命

### 需要主程补的配置字段方向

建议主动技能至少支持这些配置项：
- `Priority`
  - 技能释放优先级
- `Cooldown`
  - 冷却回合
- `MPCost`
  - 法力消耗
- `TriggerType`
  - 触发类型
- `TriggerValue`
  - 触发参数
- `TargetRule`
  - 目标选择规则
- `CastLimit`
  - 单场释放上限，可选
- `SkillTags`
  - 用于标记输出、治疗、护盾、控制、终结、破甲、AOE 等用途

### 技能触发与目标规则显示文案的本地化要求

- `TriggerType`、`TargetRule` 在配置中继续保留稳定枚举值，不要直接改成中文字段值
- 技能详情页、功法面板、奖励预览等展示层，不要把这些枚举值直接显示给玩家
- 需要由运行时把 `TriggerType`、`TargetRule` 映射到 `Localization.csv` 的 key，再取当前语言文本
- 带参数的条件文案，例如百分比、状态名、减益名，应支持 `string.Format` 这类参数替换
- 这样策划只需要维护：
  - 配置里的稳定枚举
  - `Localization.csv` 里的中英文本
- 当前建议的 key 命名已经补到：
  - `skill.trigger.*`
  - `skill.target.*`
- 目标是：
  - 中文版显示中文
  - 英文版显示英文
  - 后续新增触发类型时，只需要补枚举映射和本地化 key，不要再在代码里写死中英文文案

### 触发条件建议第一版至少支持这些类型

#### 输出类
- 默认可放
- 敌方数量大于等于 `X`
- 目标生命低于 `X%`
- 目标生命高于 `X%`
- 目标有护盾时
- 目标有增益时

#### 生存类
- 自身生命低于 `X%`
- 我方生命低于 `X%`
- 自身无护盾时
- 自身处于异常状态时

#### 节奏类
- 首回合优先释放
- 冷却结束立即可放
- 敌方准备大招时
- 自身拥有某个 Buff 时
- 目标带某个 Debuff 时

### 目标选择规则建议第一版至少支持
- 敌方生命最低
- 敌方生命最高
- 敌方攻击最高
- 随机敌方
- 全体敌方
- 自身
- 我方生命最低
- 我方全体

### 技能策略性的目标

策划希望通过这套系统达成：
- 玩家不是单纯“拿到更多技能”，而是在“配置自己的战斗逻辑”
- 自动战斗也能体现构筑差异
- 技能强度不只看倍率，还看触发条件与适配场景
- 某些技能即使不是数值最高，也能因为条件特化而有上场价值

### 技能设计原则

后续配技能时，希望遵守这些原则：
- 尽量不要做纯下位替代技能
- 每个技能尽量有明确定位
- 输出、AOE、续航、保命、控制、破甲、收割尽量职责明确
- 一个技能可以不“泛用最强”，但应在特定场景有明确价值

### 当前版本暂不做的内容

这些先记住方向，但当前版本可以不急着首批接入：
- 遗忘技能换资源
- 战斗中临时切换技能栏
- 太复杂的多段战术脚本
- 玩家手动指定每个技能的详细开关面板

### 当前版本建议优先实现的最小闭环

建议先做这一版：
- 技能库可持有多个技能
- 角色可装备最多 `3` 个主动技能
- 每个技能支持基础触发条件
- 每回合从可用技能中按优先级选一个释放
- 如果没有可释放技能，则普攻

只要先把这套跑通，策划就已经可以开始通过配表调整：
- 技能定位
- 技能竞争关系
- 自动战斗的行为差异
- 关卡与 Boss 对技能搭配的要求

## 二、战斗公式需要配置化

### 当前问题

关键战斗常量和公式如果继续写死，后面 `100` 关长线、五行怪特征、Boss 机制、事件 buff 都会越来越难调。

### 需要支持的配置项

- 基础伤害系数
- 防御折算方式
- 技能倍率系数
- 治疗系数
- 护盾系数
- 暴击相关系数
- 易伤 / 破甲 / 受疗加成等乘区或加区
- 五行克制系数上限
- `圣元素` 豁免规则
- Boss 特殊系数上限

### 策划希望的结果

- 做一个统一的战斗公式配置源
- 后续策划主要调配置，不再频繁找主程改公式常量

## 三、五行关系与圣元素需要正式接入

### 当前开发目标

- 普通怪 / 精英怪保留 `金木水火土`
- 五行怪的特征要真实生效，不只是名字和说明
- 支持特殊元素标签，例如 `圣元素`
- 特殊元素与普通五行的关系应由配置决定，不要写死在代码里
- Boss、精英、普通怪到底使用什么元素，应交给策划配置

### 需要主程补的能力

- 元素关系统一判定入口
- 普通元素之间的克制配置化
- `圣元素` 中性判定配置化
- 后续支持扩展特殊标签，例如：`魔 / 仙 / 混沌`

### 策划目的

- 保留五行怪辨识度
- 避免某个主角被某章 Boss 克得过死
- 后续如果要做特殊首领，不需要再重写整套元素判定

## 四、敌人与 Boss 阵容要走配置

### 当前问题

如果关卡已经扩成 `100` 关，但战斗阵容仍然写死，章节体验和五行差异会非常弱。

### 需要支持的配置能力

- 普通战 / 精英战 / Boss 敌人池配置
- 按章节切敌组
- 按关卡段切敌组
- 按节点类型切敌组
- Boss 单独配置
- 支持章节内不同五行怪混编
- Boss 单独指定 `圣元素标签 + 章节机制模板`

### 策划目的

- 章节感真正落到玩法上
- 五行怪不是纯文案差异
- Boss 是章节玩法的收束，不只是更大数值怪

### 当前接入约定

- 敌方编组统一走 `EnemyEncounter.csv`
- 运行时按 `StageFrom / StageTo + NodeType` 选编组
- 敌方元素、特殊标签、Boss 模板应允许在编组表里单独覆盖，不要在代码里预设固定结果

## 五、怪物特征要落成真实机制

### 当前策划定义

- 金：压制、破甲、点杀
- 木：回复、缠绕、拖战
- 水：减速、冻结、护盾、节奏干扰
- 火：爆发、AOE、灼烧
- 土：高防、护盾、反击、持久战

### 需要主程补的能力

- 至少让每种特征有一到两个真实可运行的效果承载
- 这些效果尽量走统一配置结构，不要每个怪都手写特例逻辑

### 策划目的

- 玩家能明显感受到“五行怪打起来不一样”
- 章节差异来自机制，而不是只来自数值高低

## 六、事件收益与战斗收益继续拆开配置

### 当前目标

- 战斗收益、非战斗收益、事件消耗、事件奖励、功法奖励档位，尽量分开配置
- 长线关卡后，中后期收益节奏不能只靠代码公式硬推

### 需要主程支持

- 战斗收益支持按关卡或章节配置
- 非战斗节点收益支持按 `Profile` 配置
- 固定服务点与随机机缘分开结算
- 事件冷却和服务点机制尽量配置化

## 七、配置校验工具继续补强

### 当前建议增加的校验项

- `StageNode.csv` 是否完整覆盖当前策划配置的关卡范围
- Boss 关卡是否都已正确配置到 `StageNode.csv`
- `EventProfile` 是否都能在 `EventOption.csv` 找到 `1 / 2 / 3`
- `Localization key` 是否存在
- 技能效果字段是否完整
- 元素标签是否合法
- `圣元素 Boss` 是否误吃五行克制
- 敌人池 / Boss 配置是否缺档

## 八、配置优先级约定

以后新增功能时，优先级建议是：
1. 先定义策划字段和配置结构
2. 再接运行时读取
3. 再补编辑器导表与校验
4. 最后保留必要默认值

### 目标

- 数值和文字尽可能走配置给策划维护
- 主程主要负责：框架、读取、校验、运行时结算
- 策划主要负责：填表、调数、改文案、迭代节奏

## 十、建议补一个“轻量目标系统”

### 当前策划结论

建议补的是“轻量目标系统”，不是传统 RPG 的复杂任务树。

目的主要有三点：
- 给系统人格化的引路人一个稳定的发话和引导入口
- 让玩家始终知道当前该做什么，不至于只剩抽象地图推进
- 把剧情推进、玩法引导、阶段目标统一到一套结构里

当前不建议首批就做：
- 多 NPC 并行任务链
- 复杂支线网
- 大量可失败任务分支
- 满屏任务列表

### 目标系统希望解决的问题

当前版本里，很多引导是分散的：
- 有些在剧情对白里
- 有些在系统提示里
- 有些只能靠玩家自己猜当前目标

这样会导致：
- 引路人像在说话，但没有稳定的“任务承接物”
- 玩家容易不知道当前阶段的短期目标
- 剧情推进和玩法推进显得分离

因此希望补一套足够轻、但可配置的目标系统。

### 第一版建议结构

第一版建议只做三层：

- `主目标`
  - 当前阶段最核心的推进目标
  - 例如：抵达下一处异变节点、击败本章镇守者、完成一次整备

- `阶段目标`
  - 当前阶段下的辅助进度项
  - 例如：完成一次功法配置、经过一次休整点、获得一次精英战奖励

- `引导建议`
  - 不算正式任务完成条件，更像系统建议
  - 例如：当前功法位未配置、建议先前往休整点、当前战力不足不建议强推

这样做的好处是：
- 前端显示不会太重
- 很适合当前版本体量
- 后续可以自然升级成更完整的任务系统

### 当前策划希望的前台表现

前台不一定直接叫“任务”，可以更贴世界观，例如：
- `当前目标`
- `天机指引`
- `试炼目标`
- `演算指引`

其中第一版最建议先用：
- `当前目标`

这样理解成本最低，后续再根据剧情包装替换成更世界内的叫法。

### 与系统人格引路人的关系

这个系统不是独立于剧情的 UI 功能，而是引路人“系统人格”的重要表达载体。

理想状态下：
- 前期玩家会觉得是系统在发任务、给建议
- 中期玩家会感到它不只是中立工具，而是在评估自己的选择
- 后期玩家会意识到，这些目标与引导本身就是试炼和筛选的一部分

也就是说：
- 目标系统承接功能引导
- 也承接剧情气质
- 是系统人格落地的关键接口之一

### 第一版建议最少支持的配置能力

建议至少支持一张目标配置表，例如后续可命名为：
- `Objective.csv`
或
- `GoalConfig.csv`

建议第一版字段方向至少包括：
- `Id`
  - 目标 ID
- `Group`
  - 所属阶段或章节组
- `Type`
  - 目标类型，例如：`Main`、`Stage`、`Hint`
- `TitleKey`
  - 标题文案 key
- `ContentKey`
  - 描述文案 key
- `ConditionType`
  - 完成条件类型
- `ConditionParam`
  - 条件参数
- `TargetValue`
  - 目标值
- `NextId`
  - 完成后切到哪个目标
- `AutoTrack`
  - 是否自动设为当前追踪目标
- `Visible`
  - 是否显示给玩家
- `Notes`
  - 备注

### 第一版建议支持的条件类型

建议优先支持这些最常用条件：
- 到达指定关卡
- 进入指定节点类型
- 击败指定关卡敌人
- 击败指定节点类型敌人
- 获得一次技能奖励
- 完成一次技能配置
- 进入一次休整/交易/静室类节点
- 触发一次剧情节点

这样已经足够承载：
- 新手引导
- 章节推进
- 简单剧情驱动
- 系统建议

### 运行时建议

第一版运行时不需要做得太复杂，建议：
- 同时只显示 `1` 个主目标
- 可附带 `1~3` 个阶段目标
- 引导建议可以单独放在说明区，不一定做成“完成型任务”

目标更新时机建议支持：
- 开局
- 进入地图
- 进入战斗前
- 战斗胜利后
- 获得技能/装备后
- 剧情节点结束后

### 与剧情和对话系统的关系

建议目标系统与现有 `StoryNode.csv / StoryTrigger.csv` 保持松耦合：
- 剧情可以触发目标变化
- 目标完成也可以触发剧情
- 但不要把剧情节点和目标节点写死绑定成一套结构

这样后续策划能更自由地决定：
- 哪段剧情只是演出
- 哪段剧情要顺带推送新目标
- 哪个目标完成后触发下一段对白

### 第一版策划目标

这套轻量目标系统第一版主要为了做到：
- 玩家进入游戏后更有方向感
- 引路人的“系统人格”有明确的功能承接
- 剧情与玩法推进更自然地扣在一起
- 后续如果要做更完整的任务系统，不需要推倒重来

### 当前结论

当前策划建议是：
- 应该补目标系统
- 但先做轻量版
- 先服务引导、剧情承接、阶段推进
- 不要首批就做成传统复杂任务树

## 十一、装备建议采用“5品级 + 5阶级”，暂不做强化

### 当前策划结论

装备系统当前建议采用两层结构：
- `品级`
- `阶级`

并且：
- 当前版本暂时不做装备强化
- 不做 `Lv+1`
- 不做升星、洗练、突破、精炼
- 装备强度主要由“掉落时的品级和阶级”决定

### 一、品级设计

装备品级建议与技能稀有度统一，采用 `5` 档：
- `白`
- `绿`
- `蓝`
- `紫`
- `金`

如果后续前台想包装成更修仙的显示名，可以在展示层再转成：
- `凡`
- `良`
- `灵`
- `宝`
- `绝`

但内部配置建议先保持 `白 / 绿 / 蓝 / 紫 / 金`，更直观也更方便调表。

### 二、阶级设计

装备再单独拥有 `5` 个阶级，用于表示当前装备处于哪个阶段档位。

建议前台使用：
- `一阶`
- `二阶`
- `三阶`
- `四阶`
- `五阶`

阶级的作用主要是：
- 决定基础属性档位
- 决定特性数值档位
- 体现“越往后打，掉的装备自然越高阶”

### 三、品级与阶级的分工

当前策划希望两者职责拆开：

- `品级`
  - 决定稀有度
  - 决定特性层次
  - 决定是否具备构筑价值

- `阶级`
  - 决定数值阶段
  - 决定装备是否属于当前章节/阶段的掉落强度

也就是：
- 品级管“这件装备好不好”
- 阶级管“这件装备处在第几个成长阶段”

### 四、当前版本不做强化的原因

当前版本先不做装备强化，是因为：
- 玩法重点应放在路线、技能、时间分配和构筑选择
- 如果装备再叠加强化、升星、洗练，会明显拉高系统复杂度
- 当前阶段更适合让装备承担“掉落成长 + 构筑差异”，而不是额外养成负担

因此建议：
- 玩家靠打到后期自然获得更高阶装备
- 玩家靠精英、Boss、事件、商店获得更高品装备
- 而不是再额外经营一套装备养成线

### 五、掉落规则建议

当前建议是：
- 装备的`阶级`按关卡区间自动决定
- 装备的`品级`按节点类型和奖励来源决定

即：
- 越往后的关卡，掉落装备的阶级越高
- 越高质量的战斗和事件，掉落装备的品级越高

### 六、建议的掉落倾向

- 普通战
  - 当前关卡阶级的 `白 / 绿` 为主

- 精英战
  - 当前关卡阶级的 `绿 / 蓝` 为主

- Boss 战
  - 当前关卡阶级的 `蓝 / 紫` 为主，低概率 `金`

- 商店 / 熔炉 / 特殊机缘
  - 作为定向获取高品装备的重要来源

### 七、品级层次的设计建议

当前建议不同品级承担不同职责：

- `白`
  - 纯基础属性装
  - 主要承担前期保底

- `绿`
  - 基础属性更高
  - 可开始带轻特性

- `蓝`
  - 开始具备明确的小构筑价值

- `紫`
  - 具备明显的构筑价值
  - 可作为中后期核心装备

- `金`
  - 作为高稀有核心装
  - 具备强特性或关键 build 价值

### 八、建议的开发支持方向

建议后续装备配置结构至少支持：
- `Quality`
  - 品级
- `Tier`
  - 阶级
- `DropStageMin / DropStageMax`
  - 掉落阶级生效区间，或由掉落逻辑统一映射
- `BaseStats`
  - 当前阶级基础属性
- `FeatureType / FeatureValue`
  - 装备特性类型与数值

如果后续装备条目仍然继续由 `Equipment.csv` 维护，建议至少补足：
- 品级字段
- 阶级字段
- 掉落阶段区间或掉落档位字段

### 九、当前策划目标

这套设计第一版主要想做到：
- 玩家后期自然会拿到更高阶装备
- 玩家通过挑战与选择去搏更高品装备
- 装备有构筑差异，但不额外引入复杂强化线
- 与当前技能系统一样，重点放在“选择什么装备”，而不是“怎么养一件装备”

### 当前结论

当前策划正式建议是：
- 装备采用 `5品级 + 5阶级`
- 品级：`白 / 绿 / 蓝 / 紫 / 金`
- 阶级：`一阶 / 二阶 / 三阶 / 四阶 / 五阶`
- 阶级按关卡区间掉落
- 品级按奖励来源和挑战强度分层
- 当前版本先不做装备强化系统
