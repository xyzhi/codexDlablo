# 策划给开发的文档

这份文档以后固定用于记录：
- 策划确认后的开发补充需求
- 当前版本尚未接完、但后续必须补齐的系统能力
- 希望主程优先做成配置化的运行时与工具支持

这份文档是策划写给开发看的固定文档。
如果是开发反向告诉策划“当前哪些字段已接入、怎么配、有什么约束”，统一看：
`Docs/策划配置总说明.md`

## 当前版本背景

- 当前主流程关卡数按 `StageNode.csv` 配置读取
- Boss 关卡位置按 `StageNode.csv` 配置
- 最终首领关由策划在 `StageNode.csv` 中配置
- 战斗公式由 `BattleFormula.csv` 配置读取
- 元素关系与特殊元素标签应由 `ElementRelation.csv` 配置读取
- 敌方编组、元素覆盖、Boss 模板不应写死在代码里，应由策划配置决定
- 当前数值和文案尽量希望继续走配置给策划维护

## 一、技能系统需要改成“技能库全收 + 上场限3 + 条件触发施法”

### 当前策划结论

技能掉落方案采用：
- 掉落技能可以全部收入技能库
- 不做“背包满3个就必须丢旧技能”的方案
- 当前版本暂时不做“遗忘技能换资源”功能
- 角色上场时最多装备 `3` 个主动技能
- 战斗中不采用“按回合固定轮放技能”的机制

### 不采用固定轮放的原因

如果按回合轮放：
- 垃圾技能会强行占行动轴
- 玩家拿到新技能不一定更爽，可能反而把循环打坏
- 自动战斗会显得很蠢
- 构筑策略会退化成“尽量别拿废技能”

因此当前策划希望改成：
- 技能不是轮流放
- 而是每回合从“当前可释放的已装备技能”里，自动选一个最该放的技能

### 目标机制

建议战斗施法流程如下：
1. 每回合角色行动时，先检查已装备的主动技能
2. 筛出当前满足释放条件的技能
3. 按优先级选出一个技能释放
4. 如果没有任何技能满足条件，则普攻

也就是：
- 普攻永远作为兜底动作
- 主动技能通过条件触发和优先级竞争行动机会
- 被动技能/触发技则走独立触发逻辑

### 技能结构建议

建议把技能分成三层：
- 普攻：默认兜底，永远存在
- 主动技能：满足条件时自动释放，角色最多装备 `3` 个
- 被动/触发技：满足指定事件时自动触发，不占主动技能栏

同时建议：
- 不要只靠一个 `EffectType` 字符串分支继续硬扩
- 技能本体与技能效果尽量分表维护，建议固定为 `Skill.csv + SkillEffect.csv`
- 一个技能应允许挂多个效果，不要强制只能单效果

### 技能库与技能栏规则

当前策划希望：
- 玩家可以拥有完整技能库
- 但战斗前只能从技能库里选择最多 `3` 个主动技能上场
- 技能栏调整应放在非战斗节点进行，例如：休整点、洞府、特定事件、准备界面
- 不建议允许战斗中随时换技能

这样玩家的策略重点就会变成：
- 这关该带哪 `3` 个技能
- 打 Boss 前要不要调整技能栏
- 当前构筑更偏输出、续航、控制还是保命

### 需要主程补的配置字段方向

建议主动技能至少支持这些配置项：
- `Priority`
  - 技能释放优先级
- `Cooldown`
  - 冷却回合
- `MPCost`
  - 法力消耗
- `TriggerType`
  - 触发类型
- `TriggerValue`
  - 触发参数
- `TargetRule`
  - 目标选择规则
- `CastLimit`
  - 单场释放上限，可选
- `SkillTags`
  - 用于标记输出、治疗、护盾、控制、终结、破甲、AOE 等用途

### 触发条件建议第一版至少支持这些类型

#### 输出类
- 默认可放
- 敌方数量大于等于 `X`
- 目标生命低于 `X%`
- 目标生命高于 `X%`
- 目标有护盾时
- 目标有增益时

#### 生存类
- 自身生命低于 `X%`
- 我方生命低于 `X%`
- 自身无护盾时
- 自身处于异常状态时

#### 节奏类
- 首回合优先释放
- 冷却结束立即可放
- 敌方准备大招时
- 自身拥有某个 Buff 时
- 目标带某个 Debuff 时

### 目标选择规则建议第一版至少支持
- 敌方生命最低
- 敌方生命最高
- 敌方攻击最高
- 随机敌方
- 全体敌方
- 自身
- 我方生命最低
- 我方全体

### 技能策略性的目标

策划希望通过这套系统达成：
- 玩家不是单纯“拿到更多技能”，而是在“配置自己的战斗逻辑”
- 自动战斗也能体现构筑差异
- 技能强度不只看倍率，还看触发条件与适配场景
- 某些技能即使不是数值最高，也能因为条件特化而有上场价值

### 技能设计原则

后续配技能时，希望遵守这些原则：
- 尽量不要做纯下位替代技能
- 每个技能尽量有明确定位
- 输出、AOE、续航、保命、控制、破甲、收割尽量职责明确
- 一个技能可以不“泛用最强”，但应在特定场景有明确价值

### 当前版本暂不做的内容

这些先记住方向，但当前版本可以不急着首批接入：
- 遗忘技能换资源
- 战斗中临时切换技能栏
- 太复杂的多段战术脚本
- 玩家手动指定每个技能的详细开关面板

### 当前版本建议优先实现的最小闭环

建议先做这一版：
- 技能库可持有多个技能
- 角色可装备最多 `3` 个主动技能
- 每个技能支持基础触发条件
- 每回合从可用技能中按优先级选一个释放
- 如果没有可释放技能，则普攻

只要先把这套跑通，策划就已经可以开始通过配表调整：
- 技能定位
- 技能竞争关系
- 自动战斗的行为差异
- 关卡与 Boss 对技能搭配的要求

## 二、战斗公式需要配置化

### 当前问题

关键战斗常量和公式如果继续写死，后面 `100` 关长线、五行怪特征、Boss 机制、事件 buff 都会越来越难调。

### 需要支持的配置项

- 基础伤害系数
- 防御折算方式
- 技能倍率系数
- 治疗系数
- 护盾系数
- 暴击相关系数
- 易伤 / 破甲 / 受疗加成等乘区或加区
- 五行克制系数上限
- `圣元素` 豁免规则
- Boss 特殊系数上限

### 策划希望的结果

- 做一个统一的战斗公式配置源
- 后续策划主要调配置，不再频繁找主程改公式常量

## 三、五行关系与圣元素需要正式接入

### 当前开发目标

- 普通怪 / 精英怪保留 `金木水火土`
- 五行怪的特征要真实生效，不只是名字和说明
- 支持特殊元素标签，例如 `圣元素`
- 特殊元素与普通五行的关系应由配置决定，不要写死在代码里
- Boss、精英、普通怪到底使用什么元素，应交给策划配置

### 需要主程补的能力

- 元素关系统一判定入口
- 普通元素之间的克制配置化
- `圣元素` 中性判定配置化
- 后续支持扩展特殊标签，例如：`魔 / 仙 / 混沌`

### 策划目的

- 保留五行怪辨识度
- 避免某个主角被某章 Boss 克得过死
- 后续如果要做特殊首领，不需要再重写整套元素判定

## 四、敌人与 Boss 阵容要走配置

### 当前问题

如果关卡已经扩成 `100` 关，但战斗阵容仍然写死，章节体验和五行差异会非常弱。

### 需要支持的配置能力

- 普通战 / 精英战 / Boss 敌人池配置
- 按章节切敌组
- 按关卡段切敌组
- 按节点类型切敌组
- Boss 单独配置
- 支持章节内不同五行怪混编
- Boss 单独指定 `圣元素标签 + 章节机制模板`

### 策划目的

- 章节感真正落到玩法上
- 五行怪不是纯文案差异
- Boss 是章节玩法的收束，不只是更大数值怪

### 当前接入约定

- 敌方编组统一走 `EnemyEncounter.csv`
- 运行时按 `StageFrom / StageTo + NodeType` 选编组
- 敌方元素、特殊标签、Boss 模板应允许在编组表里单独覆盖，不要在代码里预设固定结果

## 五、怪物特征要落成真实机制

### 当前策划定义

- 金：压制、破甲、点杀
- 木：回复、缠绕、拖战
- 水：减速、冻结、护盾、节奏干扰
- 火：爆发、AOE、灼烧
- 土：高防、护盾、反击、持久战

### 需要主程补的能力

- 至少让每种特征有一到两个真实可运行的效果承载
- 这些效果尽量走统一配置结构，不要每个怪都手写特例逻辑

### 策划目的

- 玩家能明显感受到“五行怪打起来不一样”
- 章节差异来自机制，而不是只来自数值高低

## 六、事件收益与战斗收益继续拆开配置

### 当前目标

- 战斗收益、非战斗收益、事件消耗、事件奖励、功法奖励档位，尽量分开配置
- 长线关卡后，中后期收益节奏不能只靠代码公式硬推

### 需要主程支持

- 战斗收益支持按关卡或章节配置
- 非战斗节点收益支持按 `Profile` 配置
- 固定服务点与随机机缘分开结算
- 事件冷却和服务点机制尽量配置化

## 七、配置校验工具继续补强

### 当前建议增加的校验项

- `StageNode.csv` 是否完整覆盖当前策划配置的关卡范围
- Boss 关卡是否都已正确配置到 `StageNode.csv`
- `EventProfile` 是否都能在 `EventOption.csv` 找到 `1 / 2 / 3`
- `Localization key` 是否存在
- 技能效果字段是否完整
- 元素标签是否合法
- `圣元素 Boss` 是否误吃五行克制
- 敌人池 / Boss 配置是否缺档

## 八、配置优先级约定

以后新增功能时，优先级建议是：
1. 先定义策划字段和配置结构
2. 再接运行时读取
3. 再补编辑器导表与校验
4. 最后保留必要默认值

### 目标

- 数值和文字尽可能走配置给策划维护
- 主程主要负责：框架、读取、校验、运行时结算
- 策划主要负责：填表、调数、改文案、迭代节奏
