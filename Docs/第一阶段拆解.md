# 第一阶段拆解

## 阶段目标

第一阶段的目标不是把所有系统做完，而是做出一个可以反复验证的最小可玩版本。

完成标准：

1. 进入游戏后能稳定显示 UGUI 页面
2. 可以从主菜单进入战斗页
3. 战斗页能用真实数据生成我方和敌方单位
4. 双方可以自动战斗并结算胜负
5. 界面能看到基础战斗信息和结果

---

## 工作范围

第一阶段只覆盖四块内容：

1. 工程结构整理
2. 核心数据结构建立
3. 自动战斗最小原型
4. 基础战斗 UI 接入

不在第一阶段内的内容：

- 地图探索
- 掉落与装备
- 三选一技能奖励
- 局外成长
- 存档
- 大量美术和表现优化

---

## 任务拆解

### A. 工程结构整理

目标：

- 让项目目录、命名和入口统一，后面扩展时不乱

子任务：

1. 统一 `Assets/Scripts` 目录结构
2. 区分 `Config / Core / Battle / Runtime / UI / Editor`
3. 修复当前 UGUI 启动与显示问题
4. 增加统一的日志输出入口
5. 整理命名规范，避免后续 `UIBattlePage / BattlePage / BattlePanel` 混用

建议目录：

```txt
Assets/Scripts/
  Config/
  Core/
  Battle/
  Runtime/
  UI/
  Editor/
```

验收标准：

- 打开项目后无基础结构混乱问题
- 运行场景后 UI 能稳定出现

---

### B. 核心数据结构建立

目标：

- 为战斗和 UI 提供统一的数据模型

子任务：

1. 明确静态配置和运行时数据的边界
2. 建立角色配置模型
3. 建立敌人配置模型
4. 建立技能配置模型
5. 建立战斗单位运行时模型
6. 建立战斗结果模型

建议数据模型：

- `CharacterConfig`
- `EnemyConfig`
- `SkillConfig`
- `BattleUnitRuntime`
- `BattleTeamRuntime`
- `BattleResult`

建议字段方向：

- 角色：ID、名称、灵根、定位、HP、ATK、DEF、MP、初始技能
- 敌人：ID、名称、类型、基础属性、技能池
- 技能：ID、名称、五行、消耗、目标类型、效果类型、数值参数
- 战斗单位：当前血量、当前MP、是否死亡、技能冷却或释放条件

验收标准：

- 代码里已经不需要直接读裸字符串配置打战斗
- UI 和战斗都能依赖统一对象

---

### C. 自动战斗最小原型

目标：

- 先做出不依赖手操的自动战斗闭环

子任务：

1. 建立 `BattleManager`
2. 建立我方与敌方队伍生成流程
3. 实现基础战斗时钟或回合驱动
4. 实现普通攻击结算
5. 实现 MP 消耗与技能释放
6. 实现基础 AI 决策
7. 实现死亡判定
8. 实现战斗结束判定
9. 输出战斗日志

建议第一版规则：

- 每个单位按固定节奏行动
- 优先释放可释放技能
- 否则进行普通攻击
- 伤害先采用简单公式，先不做复杂克制
- 先只支持单体伤害和群体伤害两类技能

建议最小伤害公式：

```txt
damage = max(1, attacker.ATK - target.DEF)
```

建议第一版 AI：

```txt
如果有可释放技能 -> 放第一个可释放技能
否则 -> 普通攻击
```

验收标准：

- 一场战斗可以自动开始、推进、结束
- 能明确得出胜利或失败结果

---

### D. 基础战斗 UI 接入

目标：

- 让战斗原型可以被观察、调试和验证

子任务：

1. 在战斗页展示我方单位列表
2. 在战斗页展示敌方单位列表
3. 显示 HP、MP、角色名、状态
4. 增加战斗日志区域
5. 增加战斗开始按钮
6. 增加返回主菜单按钮
7. 增加战斗结果弹窗
8. 增加撤退按钮占位

建议显示模块：

- 顶部：标题和战斗状态
- 中部：我方与敌方单位卡片
- 底部：日志区和操作按钮

验收标准：

- 可以从 UI 上看懂战斗发生了什么
- 能看见角色血量变化和最终结果

---

## 推荐执行顺序

按下面顺序做，返工会最少：

1. 先修 UI 框架和工程结构
2. 再建立核心数据结构
3. 再做自动战斗逻辑
4. 最后把战斗逻辑接到 UI

原因：

- UI 不稳定，后续没法验证
- 数据结构不先定，战斗和 UI 会各写一套
- 战斗逻辑先跑起来，UI 才知道该展示什么

---

## 具体待办清单

### 第一步：修框架

- 检查当前 UI 自动启动是否正常
- 确保进入 Play 后可以看到主菜单
- 确保按钮点击有效
- 确保弹窗和 Toast 正常显示

### 第二步：补齐战斗数据

- 新建敌人配置表
- 新建技能配置表
- 建立运行时单位类
- 建立配置加载与转换流程

### 第三步：实现战斗循环

- 创建战斗管理器
- 生成测试双方阵容
- 加入普通攻击逻辑
- 加入技能释放逻辑
- 加入战斗结果输出

### 第四步：把战斗挂到界面

- 战斗页绑定战斗管理器
- 动态刷新角色数据
- 输出战斗日志
- 胜负后显示结果弹窗

## 风险点

### 风险 1：UI 框架不稳定

影响：

- 会直接阻塞后续所有验证工作

应对：

- 第一优先级先把启动、显示、按钮响应跑通

### 风险 2：数据结构反复改

影响：

- 会导致战斗逻辑和 UI 一直返工

应对：

- 第一阶段只先支持最小字段，不一开始追求完美

### 风险 3：战斗规则做太复杂

影响：

- 核心玩法验证速度会变慢

应对：

- 第一版只做最简单的自动战斗规则
