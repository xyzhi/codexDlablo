# 《五行行者》开发计划

## 目标

基于当前策划大纲，先做出一个可持续迭代的单机肉鸽原型。开发顺序以“先验证核心乐趣，再补系统，再扩内容”为原则，优先保证：

1. 战斗能跑起来
2. 探索和战斗能形成一局完整流程
3. 配表和UI框架能支撑后续快速迭代

---

## 当前阶段判断

目前项目已经有：

- 角色CSV配置雏形
- CSV 转 JSON 的配置流程
- 一套最小 UGUI 框架雏形

目前项目还缺：

- 真正可玩的战斗循环
- 角色/敌人运行时数据结构
- 技能释放逻辑
- 地图探索流程
- 掉落与养成闭环
- UI 和数据的正式联动

所以接下来最合适的路线不是先堆内容，而是先搭“主循环骨架”。

---

## 第一阶段：打通最小可玩版本

### 1. 整理基础工程结构

需要做的事情：

- 统一 `Scripts` 目录结构，按 `Config / Core / Battle / Map / UI / Runtime` 划分
- 确认 UI 框架稳定可运行，修正当前启动与显示问题
- 建立基础日志输出与调试入口
- 建立通用常量、枚举、工具类

产出：

- 清晰的代码目录
- 可稳定进入主菜单和战斗页的基础框架

### 2. 定义核心数据结构

需要做的事情：

- 定义角色静态配置结构
- 定义敌人静态配置结构
- 定义技能/功法静态配置结构
- 定义运行时战斗单位数据结构
- 定义局内成长数据结构

产出：

- `Character / Enemy / Skill / BattleUnit / RunData` 等基础模型
- 为后续战斗和UI提供统一数据来源

### 3. 先做自动战斗原型

需要做的事情：

- 搭建战斗管理器
- 生成我方与敌方战斗单位
- 实现基础攻击流程
- 实现生命值、攻击、防御、MP 等核心属性计算
- 实现最简单的 AI 决策：有技能放技能，否则普通攻击
- 实现胜负判定

产出：

- 一场能自动打完的战斗 Demo
- 可在界面中看到基础战斗结果

### 4. 接入基础战斗 UI

需要做的事情：

- 显示角色与敌人的基础信息
- 显示血量、MP、状态文本
- 显示战斗日志
- 显示战斗结果面板
- 接入撤退按钮和基础交互

产出：

- 一个可观察、可调试、可反复测试的战斗界面

---

## 第二阶段：做出完整单局闭环

### 5. 建立探索地图流程

需要做的事情：

- 设计地图节点数据结构
- 实现线性或弱分支的节点推进
- 支持战斗节点、事件节点、奖励节点、商店节点占位
- 记录当前进度与章节信息

产出：

- 玩家可以从起点一路推进到 Boss 的地图流程

### 6. 接入战斗结算与奖励

需要做的事情：

- 按策划规则结算灵石奖励
- 支持撤退成功/失败逻辑
- 战后弹出三选一功法界面
- 将选择结果记录到当前局数据中

产出：

- 探索 -> 战斗 -> 结算 -> 选技能 -> 继续探索 的完整循环

### 7. 实现功法系统第一版

需要做的事情：

- 建立技能配置表
- 实现五行功法分类
- 实现技能等级提升
- 实现多灵根降低单功法等级上限的规则
- 实现常见效果类型：单体伤害、群体伤害、治疗、护盾、DOT、控制

产出：

- 可构筑、可成长、能明显区分流派的功法系统

---

## 第三阶段：丰富内容与可玩性

### 8. 建立敌人与章节配置

需要做的事情：

- 配置普通敌人、精英敌人、Boss
- 按地图章节配置固定强度
- 做敌人阵容组合规则
- 支持敌人技能差异化

产出：

- 具备章节推进感的战斗内容

### 9. 建立卡牌与掉落系统

需要做的事情：

- 明确角色卡和装备卡的数据结构
- 实现战斗掉落
- 实现稀有度与概率控制
- 实现卡牌背包展示

产出：

- 有收集反馈的局外或局内卡牌系统雏形

### 10. 建立装备系统

需要做的事情：

- 定义装备基础属性
- 定义法术词缀配置
- 实现装备掉落生成
- 实现穿戴与卸下逻辑
- 在角色面板中显示装备收益

产出：

- 基础暗黑式掉装体验

---

## 第四阶段：体验优化与数值平衡

### 11. 做基础数值验证

需要做的事情：

- 拉通角色成长曲线
- 校验敌方强度曲线
- 校验技能强度区间
- 校验战斗时长
- 校验掉落和构筑节奏

产出：

- 一版能稳定玩 10 到 20 分钟一局的数值框架

### 12. 做 UI 和交互打磨

需要做的事情：

- 统一页面风格
- 增加转场、反馈、按钮态
- 优化弹窗层级与通用组件
- 优化战斗信息展示密度

产出：

- 一版不只是“能用”，而是“能顺畅游玩”的界面体验

### 13. 做存档与局外数据

需要做的事情：

- 本地存档
- 玩家拥有卡牌记录
- 解锁进度
- 基础设置项

产出：

- 可反复游玩与迭代测试的工程基础

---

## 推荐开发顺序

如果由我来推进，我会按下面顺序做：

1. 修好现有 UGUI 框架，让页面稳定显示
2. 建立角色、敌人、技能三套配置结构
3. 做自动战斗最小原型
4. 把战斗 UI 接上真实数据
5. 做战后奖励和三选一技能
6. 做探索地图和节点推进
7. 做敌人章节化配置
8. 做装备和掉落
9. 做卡牌背包与整理
10. 做整体数值调优和体验优化

---

## 每周推进建议

### 第 1 周

- 修 UI 框架
- 建战斗基础类
- 跑通自动战斗

### 第 2 周

- 做技能系统第一版
- 做战斗表现和结果界面
- 跑通战斗结算

### 第 3 周

- 做地图节点推进
- 跑通单局循环
- 初步接入奖励与成长

### 第 4 周

- 扩敌人和技能内容
- 做装备原型
- 开始数值平衡

---

## 当前最优先待办

接下来立刻应该做的不是继续铺新文件，而是：

1. 先把当前 UGUI 框架修到运行可见
2. 再把战斗最小原型搭出来
3. 然后用真实配置数据驱动界面

原因：

- UI 不稳定，后面所有系统都不好验证
- 没有战斗原型，策划大纲无法快速验证核心乐趣
- 先打通主循环，后续加角色、技能、装备才不会返工太大

---

## 里程碑定义

### 里程碑 M1：战斗 Demo

标准：

- 进入游戏后可打开主菜单
- 可进入战斗页
- 双方自动战斗
- 可结算胜负

### 里程碑 M2：单局闭环

标准：

- 可探索节点
- 可进入战斗
- 可获得奖励
- 可选功法并继续推进

### 里程碑 M3：内容扩展版

标准：

- 有多个敌人组合
- 有多套功法流派
- 有装备与掉落
- 有基础数值平衡

---

## 备注

这份计划以“先做可玩的最小版本”为前提，不追求一开始就把所有策划点全部实现。只要核心闭环先成立，后面五行克制、卡牌收集、装备词缀、局外成长都可以逐步接进去。
